## Architecture Decisions (Section 1)

| Decision | Resolution |
|---|---|
| Scope | MVF (W1-W4) + Payment Repair (W5-W12). Disputes deferred. |
| Multi-tenancy | Single-tenant now, `tenant_id` columns from day 1. |
| Tech stack | Node.js/TypeScript + Fastify + Postgres + BullMQ + Temporal |
| Classifier | Hybrid: rule engine for known decline codes, LLM for ambiguous/novel cases |

**BullMQ / Temporal division (hard line):**
- BullMQ: transient work — webhook ingestion queue, context enrichment jobs, notification sends
- Temporal: durable action execution — retries with delayed scheduling, rollback, anything running >seconds

**MVF components (9 engineer-days):**
1. Event ingestion schema (normalized payment event, PSP adapter interface)
2. Mandate data model (see schema below)
3. Audit log schema (append-only, hash-chained)
4. Connector interface stub (`(event: NormalizedPaymentEvent) => void`)

## Error & Rescue Map (Section 2)

| Failure Mode | Rescue Strategy |
|---|---|
| Webhook not delivered | Periodic Stripe Events API poll as backstop |
| Context enrichment timeout | PSP-only classification + human review flag |
| LLM failure (hybrid classifier) | Rules-only fallback → lower confidence → escalates to human queue |
| Action execution failure | Temporal workflow retries + rollback on terminal failure |
| Mandate validation failure | Block + log to audit + surface in HITL approval queue |
| Playbook miss (no matching rule) | Escalate to human queue — never silently drop |
| Circuit breaker trigger | Degrade to manual. **Define threshold before W7:** failure rate % over time window + what "degraded" means. |

## Security & Threat Model (Section 3)

| Threat | Mitigation |
|---|---|
| Prompt injection via webhook fields | Structured JSON schema wrapping; strict LLM output validation against response schema |
| Tenant API key exposure | Key fetched per Temporal activity; never logged; key ID only in logs |
| Audit log tampering | Append-only Postgres table (app DB role `UPDATE`/`DELETE` revoked) + hash chain |

## Data Flow & Edge Cases (Section 4)

**Happy path:**
```
Stripe webhook → ACK (immediate) → enrich (async, 30s budget)
  → classify (rules; LLM if ambiguous)
  → playbook eval (IF/THEN match)
  → mandate check (policy firewall)
  → [auto-execute OR approval queue]
  → Temporal workflow (action execution)
  → outcome writeback → audit log
```

**Edge case fixes (locked):**

1. **Context not ready**: `context_readiness` flag per tenant. During initial sync, all events route to human queue with "context sync in progress" label.

2. **Retry history**: `payment_failure_history` table keyed by original payment intent ID. Every action execution writes outcome back. Playbook engine queries this before rule evaluation. Prevents infinite retries of same parameters.

3. **Mandate expires mid-action**: Temporal workflow completes with status `MANDATE_EXPIRED`, writes to audit log, surfaces in HITL queue. Never leaves workflow in limbo.

## Test Requirements (Section 7)

Must-have before W7 (auto-execution goes live):
1. **Mandate policy firewall**: unit test every scope violation combination. 100% coverage.
2. **Playbook engine**: property-based tests — random decline code + context → always valid action or human escalation. Never crashes, never silent miss.
3. **Action idempotency**: integration test — same webhook 3x → exactly one action executed.
4. **Classifier confidence floor**: unit test — confidence < 0.60 → always escalates. No exceptions.

## Per-Connector Timeout (Section 10)

10 seconds per connector (not 30s total). Each connector fails independently. Specify this in the connector abstraction interface during MVF.

---

## UX / Mandate Binding (Section 9)

Per-action autonomy toggles in the UI sidebar are mandate updates, not settings preferences. When merchant flips "outreach = auto," it writes back to the mandate object. One source of truth. If these diverge, you get autonomy incidents.

## Revised 12-Week Delivery Plan

### W1–W4: MVF

**W1–W2:**
- Normalized payment event schema (PSP-agnostic)
- Stripe webhook endpoint (idempotent by PSP event ID)
- `tenant_id` columns on all tables
- Audit log: append-only Postgres table + hash chain
- Connector interface stub with per-connector 10s timeout
- `context_readiness` flag per tenant

**W3–W4:**
- Mandate data model + policy firewall
- `payment_failure_history` table
- BullMQ queue for webhook ingestion + context enrichment jobs
- Temporal setup (local dev) — one no-op workflow to verify infra
- CI/CD pipeline (GitHub Actions → Fly.io)

Acceptance gate: can ingest a Stripe test webhook, persist it to Postgres with audit log entry, check against a stub mandate, and return a logged outcome. No classification yet.

### W5–W12: Payment Repair

**W5–W6:**
- Failure classifier v1: rule engine for known decline codes (Stripe codes)
- LLM integration for ambiguous cases (confidence score output)
- LLM fallback: on failure → rules-only, lower confidence floor
- Repair queue UI v0 (read-only list + filters)

**W7–W8:**
- Playbook engine v1 (IF/THEN rules, `payroll_day_proximity`, `retry_count` from history)
- Playbook miss → escalate to human queue
- Mandate v1: action toggles + amount/frequency thresholds
- Circuit breaker thresholds defined + implemented
- Auto-retry for safe actions (specific decline codes, small amounts)
- Mandate policy firewall tests (100% coverage)

**W9–W10:**
- Context connectors v1: at least one (Shopify or Salesforce depending on first customer)
- HITL approval UI: single/bulk approvals, mandate badge, agent reasoning
- Outcome tracking: execution success/failure, re-queue rules
- Idempotency integration tests

**W11–W12:**
- Audit export + permission matrix
- Metrics dashboards (business outcome strip: at-risk MRR, recovery rate delta, MTR, dollars recovered)
- Pilot runbook: alerts, circuit breaker thresholds, rollback procedure
- First pilot merchant: 2–4 weeks of data, publishable ROI evidence

## Success Criteria (unchanged from design doc)

For first paid pilot (8 weeks from merchant go-live):
- Recovery rate ≥ 65% on soft declines (agent-initiated, approval latency < 4h; ~40% manual baseline assumed)
- Mean time to repair ≤ 6 hours (vs. 3-5 day manual baseline)
- Zero actions executed outside mandate scope (verified via audit log)
- Audit log complete for 100% of actions
- At least one mandate object per action type, covering full pilot scope
